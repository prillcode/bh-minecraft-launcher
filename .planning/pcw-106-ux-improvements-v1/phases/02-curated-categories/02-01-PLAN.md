# Plan: 02-01 — Modrinth Bulk Project Fetch (IPC + API)

**Phase:** 02 — Curated Mod Categories + Shader Suggestions
**Work:** pcw-106 ux-improvements-v1

## Context

`ModrinthAPI` has `getProject(slug)` (single fetch) but not a bulk fetch.
Modrinth exposes `GET /v2/projects?ids=["slug1","slug2"]` — returns an array of projects in one call.
The renderer will call this with a hardcoded slug list when a category chip is clicked.
No new IPC channel is needed on the shader side — the renderer can call the same `mods:get-projects`
channel since it returns the same `ModrinthProject[]` shape and shaders are just projects.

## Steps

### Step 1 — Add `getProjects()` to `ModrinthAPI`

**File:** `src/core/mods/modrinth-api.ts`

Add after the existing `getProject()` method:

```ts
/**
 * Fetch multiple projects by slug or ID in a single request.
 * Modrinth: GET /v2/projects?ids=["slug1","slug2"]
 */
async getProjects(slugs: string[]): Promise<ModrinthProject[]> {
  if (slugs.length === 0) return [];
  return this.client
    .get('projects', { searchParams: { ids: JSON.stringify(slugs) } })
    .json<ModrinthProject[]>();
}
```

**Verify:** TypeScript happy, return type is `ModrinthProject[]`.

---

### Step 2 — Add `mods:get-projects` IPC handler

**File:** `src/main/ipc-handlers.ts`

Add after the `mods:search` handler:

```ts
ipcMain.handle('mods:get-projects', async (_e, slugs: string[]) => {
  return modrinthAPI.getProjects(slugs);
});
```

No instance filtering here — the renderer handles loader/version compatibility display.

### Step 3 — Expose in preload

**File:** `src/main/preload.ts`

